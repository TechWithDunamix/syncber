Medical coding and billing are where clinical documentation meets financial reimbursement. EHR systems have transformed this interface by automating code suggestions, streamlining charge capture, and enabling electronic claims submission.

## The Coding-Billing Connection

```mermaid

    A[Clinical Encounter] --> B[Provider Documentation]
```

## Computer-Assisted Coding (CAC)

EHRs use computer-assisted coding to improve accuracy and efficiency:

```yaml
How CAC Works:
  └− Provider documents the encounter
  └− Natural language processing (NLP) extracts key terms
  └− System suggests ICD-10 and CPT codes based on documentation
  └− Coder reviews, validates, and adjusts codes
  └− Final codes submitted for claim

CAC Benefits:
  └− Improved specificity: Prompts for more specific ICD-10 codes
  └− Reduced queries: Fewer coder-to-provider clarification requests
  └− Faster coding: 20-30% reduction in coding time
  └− Higher accuracy: 5-10% improvement in coding accuracy
  └− Better reimbursement: Complete documentation supports higher E/M levels

CAC Limitations:
  └− Requires structured documentation
  └− May miss clinical context that human coder catches
  └− Initial implementation requires significant training
  └− Over-reliance can lead to cloning (same codes every visit)
```

## ICD-10 Coding in EHR

```yaml
ICD-10 Requirements:
  └− Specificity: Highest level of specificity required
  └− Laterality: Right, left, bilateral, unspecified
  └− Episode of care: Initial, subsequent, sequela (7th character)
  └− Combination codes: Some codes capture diagnosis + manifestation
  └− Excludes notes: Conditions not coded together

EHR ICD-10 Support:
  └− ICD-10 code search by keyword, condition, or body system
  └− Favorite/recent codes for quick access
  └− Specificity prompts: "This code requires laterality — right, left, or bilateral?"
  └− Medical necessity checking: Does code match the service?
  └− Edits: Age, sex, and diagnosis-procedure compatibility checks

Example — ICD-10 Specificity in EHR:
  Provider types: "Diabetes"
  EHR suggests:
    └− E11.9: Type 2 diabetes without complications
    └− E11.21: Type 2 diabetes with diabetic nephropathy
    └− E11.31: Type 2 diabetes with unspecified diabetic retinopathy
    └− E11.51: Type 2 diabetes with diabetic peripheral angiopathy
  Provider selects: E11.21 (Type 2 diabetes with nephropathy)
  EHR validates: Age-appropriate, sex-appropriate, no excluded codes
```

## CPT Coding in EHR

### Evaluation and Management (E/M) Coding

E/M codes (99202-99215) are the most common CPT codes used in outpatient settings:

```yaml
E/M Code Determination (2023 Guidelines):
  Based on ONE of two pillars:
  
  Pillar A — Medical Decision Making (MDM):
    1. Problems addressed: Number and complexity
    2. Data reviewed: Tests, records, history
    3. Risk: Management options, complications
  
  Pillar B — Time:
    Total provider time on the encounter date
    Includes: history, exam, counseling, coordination, documentation
  
  EHR Support for E/M Coding:
    └− MDM calculator: Automatically calculates MDM level based on documentation
    └− Time tracker: Records total provider time (if coding based on time)
    └− Coding alerts: "Documentation supports 99214 but billed 99213"
    └− Under-coding/over-coding warnings
    └− Audit readiness: Documentation matches code billed
```

## Charge Capture

Charge capture ensures all billable services are recorded:

```yaml
Charge Capture Methods:
  1. Automatic Charge Capture:
     └− EHR auto-generates charges based on documented services
     └− Example: Provider orders lab → charge created automatically
     └− Example: Vaccine administered → charge + diagnosis created
  
  2. Encounter-Based Charging:
     └− Provider signs encounter note → charges generated
     └− Fee schedule determines charge amounts
     └− Previous balance + new charges = total
  
  3. Point-of-Service Charging:
     └− Charges entered at the time of service
     └− Patient copay collected before visit
     └− Reduces outstanding patient balances

Charge Capture Challenges:
  └− Missed charges: Services provided but not billed (1-3% leakage)
  └− Late charges: Charges entered after claim submission
  └− Incorrect coding: Wrong codes = denial or underpayment
  └− Unbundling: Separating services that should be billed together

Charge Capture Best Practices:
  └− Same-day charge entry (before end of business day)
  └− Pre-visit charge audit (verify all services captured)
  └− Real-time charge reconciliation
  └− Regular charge lag reporting (days from service to charge entry)
```

## Claim Submission Workflow

```yaml
Step 1 — Claim Generation:
  └− Encounter documentation completed and signed
  └− Coding assigned (ICD-10, CPT, modifiers)
  └− Charges captured and verified
  └− Claim form generated (CMS-1500 or UB-04)

Step 2 — Claim Edits/Scrubbing:
  └− Clean claim checks:
       └− Required fields complete?
       └− Codes valid for date of service?
       └− Payer-specific requirements met?
       └− NPI, Tax ID, Provider ID correct?
  └− Payer-specific edits:
       └− Medical necessity check (LCD/NCD)
       └− Modifier requirements
       └− Prior authorization attached
  └− Error correction before submission

Step 3 — Claim Submission:
  └− Electronic submission via clearinghouse
  └− EDI 837 format
  └− Direct submission to some payers
  └− Confirmation number received

Step 4 — Claim Tracking:
  └− Status: Submitted, Accepted, Rejected, Denied, Paid
  └− Aging report: Days since submission
  └− Payer-specific status tracking

Step 5 — Payment Posting:
  └− ERA 835 received and auto-posted
  └− Payment applied to correct claim
  └− Adjustment applied (contractual write-off)
  └− Patient responsibility calculated

Step 6 — Denial Management:
  └− Denial reason identified and categorized
  └− Corrected and resubmitted
  └− Appeal filed if appropriate
  └− Denial rate tracked over time
```

## Revenue Cycle Management (RCM) Metrics

| Metric | Definition | Target |
|--------|------------|--------|
| **Net Collection Rate** | Amount collected / Amount billable (after adjustments) | > 95% |
| **Days in Accounts Receivable (A/R)** | Average days from charge to payment | < 35 days |
| **Clean Claim Rate** | % of claims paid on first submission | > 95% |
| **Denial Rate** | % of claims denied | < 5% |
| **Charge Lag** | Days from service to charge entry | < 2 days |
| **Adjustment Rate** | % of charges written off | < 10% |
| **Patient Responsibility Collection Rate** | % of patient balances collected | > 80% |
| **Cash Flow** | Total monthly collections | Trend positive |

## Key Takeaways

- Medical coding (ICD-10, CPT, HCPCS) translates clinical documentation into standardized codes for billing, quality measurement, and data analytics
- Computer-Assisted Coding (CAC) uses NLP to suggest codes based on provider documentation, improving accuracy and reducing coding time by 20-30%
- E/M coding is now based on Medical Decision Making (MDM) or Time — EHRs can automatically calculate MDM level from documentation
- Charge capture ensures all billable services are recorded — automatic charge capture (from documented services) reduces revenue leakage
- Claim submission follows a six-step workflow: generation → editing → submission → tracking → payment posting → denial management
- Revenue Cycle Management (RCM) metrics include net collection rate (> 95%), days in A/R (< 35), clean claim rate (> 95%), and denial rate (< 5%)
- Denial management is a critical RCM function — each denial must be analyzed for root cause and corrected
- EHR integration with coding and billing reduces manual effort, improves accuracy, and accelerates claim processing
- Poor documentation leads to denied claims, downcoding, and lost revenue — making clinical documentation quality a financial imperative
- The shift from volume-based to value-based reimbursement requires accurate coding to support risk-adjusted quality measurement
