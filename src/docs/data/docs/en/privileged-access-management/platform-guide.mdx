## CyberArk Privileged Access Manager

CyberArk is the market-leading PAM platform. This guide covers the core CyberArk components: the **Password Vault**, **Central Policy Manager (CPM)**, **Privileged Session Manager (PSM)**, and **Application Identity Manager (AIM)**.

### 1.1 Logging into the CyberArk PVWA

The **Password Vault Web Access (PVWA)** is the primary administrative interface.

![CyberArk PVWA login screen showing the vault authentication portal with username and password fields, and the CyberArk logo](https://docs.cyberark.com/Content/Resources/Images/PAS/Installation/PVWA_config_01.png)

**Steps:**

<Steps>
### Navigate to PVWA
Open a supported browser and go to the PVWA URL: `https://<PVWA_server>/PasswordVault`

### Authenticate
Log in with your vault administrator credentials. CyberArk supports:
- **Internal Vault authentication** (default)
- **LDAP / Active Directory** integration
- **RADIUS** and **SAML** for MFA

### Verify the Dashboard
After login, you'll see the **Dashboard** showing:
- Total accounts in the vault
- Active privileged sessions
- Pending access requests
- Recent policy violations
</Steps>

![CyberArk PVWA dashboard showing account statistics, active sessions, pending requests, and policy violation alerts](https://raw.githubusercontent.com/cyberark/Product-Documentation/main/PAS/Images/PVWA_Dashboard.png)

### 1.2 Onboarding a Privileged Account

<Steps>
### Navigate to Accounts
Go to **Accounts** → **Add Account** from the left navigation panel.

### Select the Platform
Choose the appropriate platform template (Windows Server, Linux, Database, Cloud, etc.). Platforms define how the account will be managed — password rotation frequency, connectivity method, and policy rules.

### Enter Account Details
Fill in:
- **Account Name** — A descriptive name (e.g., `SQL_SA_ProdDB`)
- **Address** — The target system FQDN or IP
- **Username** — The privileged account username
- **Password** — The current password
- **Safe** — The vault safe where this account will be stored

### Configure Password Management
Set the rotation schedule:
- **Automatic rotation** — Immediately after check-in
- **Periodic rotation** — Every N days (e.g., every 30 days)
- **Manual** — Rotate on demand only

### Review and Save
Verify all settings and click **Save**. The CPM will immediately attempt to verify connectivity and manage the account.
</Steps>

![CyberArk Add Account form with platform selection, account details, password management settings, and safe assignment fields](https://docs.cyberark.com/Content/Resources/Images/PAS/Accounts/AddAccount_01.png)

### 1.3 Checking Out a Password

<Steps>
### Locate the Account
Use the search bar in PVWA to find the target account by name, address, or username.

### Initiate Check-Out
Click the **Connect** or **Copy Password** button next to the account. The vault logs the checkout request with a timestamp, user identity, and reason if required.

### Set Check-Out Duration (Optional)
If the platform requires it, specify a reason ticket number and the duration for which the password will remain checked out. After this period, the password is automatically rotated by the CPM.

### Connect to the Target
Use the retrieved credentials to log into the target system. If using PSM, the connection is proxied through the PSM server and the session is recorded.
</Steps>

![CyberArk password checkout dialog showing Connect and Copy Password options with session duration and reason fields](https://docs.cyberark.com/Content/Resources/Images/PAS/Accounts/Accounts_Connect.png)

### 1.4 Managing PSM Sessions

The Privileged Session Manager (PSM) provides session isolation and recording.

<Steps>
### Configure PSM Connection
In PVWA, go to **Administration** → **PSM Configuration**. Ensure the PSM server is registered and has the target system defined.

### Initiate a PSM Session
When connecting to a managed account, select **Connect via PSM** instead of direct RDP/SSH. This routes traffic through the PSM proxy.

### Monitor Active Sessions
Go to **Dashboard** → **Active Sessions** to view all current PSM sessions. From here you can:
- View session metadata (user, target, duration)
- Terminate suspicious sessions
- Send messages to connected users

### Review Session Recordings
Go to **Reports** → **Session Recordings** to play back recorded sessions for audit and forensics.
</Steps>

![CyberArk Session Recording viewer showing recorded RDP/SSH session playback with timeline navigation and command search](https://docs.cyberark.com/Content/Resources/Images/PAS/SessionRecording/SessionRecording_02.png)

<Aside variant="tip">
**Pro Tip:** In production, use PSM for **all** administrative access to managed systems. Never allow direct RDP or SSH to servers that have PSM coverage — this defeats the purpose of session isolation.
</Aside>

## Microsoft Entra ID (Azure AD) — Privileged Identity Management

Microsoft Entra ID (formerly Azure Active Directory) is the identity and access management backbone for Microsoft cloud services. **Privileged Identity Management (PIM)** is the PAM layer within Entra ID.

### 2.1 Activating a Role in PIM

![Microsoft Entra admin center PIM interface showing eligible role assignments with activation button and justification field](https://learn.microsoft.com/en-us/entra/id-governance/media/activate-roles/pim-activate-role.png)

**Steps:**

<Steps>
### Navigate to Entra Admin Center
Go to `https://entra.microsoft.com` and sign in with your credentials. Navigate to **Identity** → **Identity Governance** → **Privileged Identity Management**.

### Select "My Roles"
Under PIM, select **My Roles** → **Azure AD roles** to see all roles you're eligible for.

### Activate a Role
Find the role you need (e.g., **Global Administrator**, **Privileged Role Administrator**) and click **Activate**.

### Provide Justification
Enter:
- **Activation reason** — A clear business justification (e.g., "Tier-0 emergency break-glass reset")
- **Duration** — How long the role is needed (default: 1 hour, max: 8 hours per policy)
- **Ticket ID** (if required by your organisation's policy)

### Authenticate with MFA
Complete an MFA challenge (Microsoft Authenticator, FIDO2 key, or other configured method). This verifies the request is legitimate.

### Wait for Activation
If **approval is required**, the request is routed to the designated approvers. They receive an email and can approve/deny through the PIM interface. If **no approval** is needed, the role activates immediately.
</Steps>

![PIM role activation approval workflow showing pending requests with approve/deny options and audit details](https://learn.microsoft.com/en-us/entra/id-governance/media/approval-workflow/pim-approval-request.png)

### 2.2 Configuring a PIM Approval Workflow

<Steps>
### Open PIM Settings
Go to **PIM** → **Azure AD roles** → **Settings** → Select a role (e.g., **Global Administrator**).

### Configure Activation Settings
Set the following policies:
- **Activation requires approval** — Enable and add one or more approvers (usually security team or role owners)
- **Require justification on activation** — Always enabled
- **Require ticket information on activation** — Optional, but recommended for audit traceability
- **Require Azure MFA on activation** — Always enabled

### Set Maximum Duration
Define the maximum activation duration (typically 1–8 hours). After expiry, the role is automatically deactivated.

### Configure Notification
Set notification recipients for:
- **Role activation** — Who gets alerted when a sensitive role is activated
- **Role assignment** — Who gets notified of new permanent assignments
</Steps>

![PIM role settings configuration page showing approval requirement toggles, notification settings, and maximum duration slider](https://learn.microsoft.com/en-us/entra/id-governance/media/pim-how-to-change-default-settings/pim-role-settings.png)

<Aside variant="info">
PIM supports **three assignment types**: **Eligible** (user must activate), **Active** (permanently assigned — avoid), and **Scheduled** (time-bound activation windows). Always use **Eligible** for administrative roles following least-privilege principles.
</Aside>

### 2.3 Creating an Access Review

Access reviews in Entra ID automate the periodic certification of privileged access.

<Steps>
### Navigate to Access Reviews
Go to **Identity Governance** → **Access Reviews** → **New access review**.

### Configure the Review
Select:
- **Review what** — Azure AD roles, group memberships, or application access
- **Scope** — All users or specific users
- **Reviewers** — Group owners, selected reviewers, or users themselves (self-review)

### Set Schedule
Configure recurrence: **One-time**, **Weekly**, **Monthly**, or **Quarterly**. Set duration (e.g., 14 days for reviewers to respond).

### Configure Auto-Apply
When a reviewer denies access, PIM can automatically remove the user from the role or group after the review period ends.

### Launch
Review the configuration and click **Start**. Reviewers will receive email notifications to complete their reviews.
</Steps>

![Access review creation wizard in Entra ID showing scope selection, reviewer assignment, and schedule configuration](https://learn.microsoft.com/en-us/entra/id-governance/media/create-access-review/create-access-review.png)

## AWS IAM — Identity and Access Management

AWS IAM manages access to Amazon Web Services. While primarily an IAM platform, it includes PAM-relevant capabilities such as roles, permission boundaries, and session policies.

### 3.1 Creating an IAM Role with Least Privilege

![AWS IAM console create role wizard showing trusted entity type selection (AWS service, Another AWS account, Web identity, SAML)](https://docs.aws.amazon.com/images/IAM/latest/UserGuide/images/roles-trusted-entity.png)

**Steps:**

<Steps>
### Open IAM Console
Sign in to the AWS Management Console and go to **IAM** → **Roles** → **Create role**.

### Select Trusted Entity
Choose the type of trusted entity:
- **AWS service** — For EC2, Lambda, ECS, etc.
- **Another AWS account** — For cross-account access
- **Web identity** — For Amazon Cognito or external IdP federation
- **SAML** — For SAML 2.0 federation (e.g., with Entra ID or Okta)

### Attach Permissions Policies
Search for and attach the required policies. Use **AWS managed policies** as a starting point, but prefer **customer managed policies** with minimal permissions for production:
- Use the **Policy Simulator** to verify effective permissions before attaching
- Apply **permission boundaries** to limit maximum permissions the role can receive

### Set Role Name and Review
Name the role (e.g., `ProdDB_ReadOnly_Role`), review the trusted entities and policies, and click **Create role**.
</Steps>

![AWS IAM role creation showing permission policy search and attachment interface with policy grant details](https://docs.aws.amazon.com/images/IAM/latest/UserGuide/images/roles-permissions.png)

### 3.2 Applying an IAM Permission Boundary

Permission boundaries are a PAM-native AWS feature that limits the maximum permissions a role or user can have — similar to a privilege ceiling.

<Steps>
### Create a Boundary Policy
Go to **Policies** → **Create policy**. Write a policy that defines the **maximum allowed permissions**:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "s3:Get*",
        "s3:List*",
        "rds:Describe*"
      ],
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": [
        "iam:*",
        "organizations:*",
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail"
      ],
      "Resource": "*"
    }
  ]
}
```

### Assign the Boundary
When creating a role (or editing an existing one), under **Permissions boundary** select **Use a permissions boundary to control the maximum role permissions** and choose the policy created above.

### Test the Boundary
Attempt to perform an action denied by the boundary (e.g., `iam:CreateUser`). AWS will deny the action even if an attached policy allows it — the boundary acts as a hard cap.
</Steps>

![IAM permission boundary assignment showing the boundary selection dropdown during role creation](https://docs.aws.amazon.com/images/IAM/latest/UserGuide/images/permissions-boundary-create-role.png)

### 3.3 Enforcing MFA for Privileged Users

<Steps>
### Create an MFA Enforcement Policy
Go to **Policies** → **Create policy** and use the following condition:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "false"
        }
      }
    }
  ]
}
```

### Attach to Privileged Groups
Attach this policy to groups containing privileged users (Admins, SecurityAudit, etc.). This denies all actions if MFA is not presented.

### Verify
Have a privileged user attempt an API call without MFA — it will be denied. The same action with MFA will succeed.
</Steps>

<Aside variant="tip">
Combine permission boundaries with **service control policies (SCPs)** in AWS Organizations for multi-account PAM. SCPs apply across all accounts in an organisation and cannot be overridden by account administrators.
</Aside>

## SailPoint IdentityIQ / IdentityNow

SailPoint provides identity governance and administration (IGA) capabilities including access certifications, role mining, and automated provisioning. It is often deployed alongside PAM platforms.

### 4.1 Performing an Access Certification Campaign

![SailPoint IdentityIQ access certification interface showing a list of entitlements pending review with certify/revoke actions](https://community.sailpoint.com/t5/image/serverpage/image-id/2905i72A9A8A9B4C6E39F/image-size/large?v=v2&px=999)

**Steps:**

<Steps>
### Navigate to Certifications
Log in to SailPoint IdentityIQ (or IdentityNow) and go to **Certifications** → **My Certifications**. You'll see a list of active and completed certification campaigns assigned to you.

### Select a Campaign
Click on the campaign name (e.g., **Quarterly Access Review — AWS Admins**). The campaign shows all users and their assigned entitlements (roles, groups, permissions) for review.

### Review Each Entitlement
For each user-entitlement pair:
- **Review the justification** — Account creation date, last access date, business role
- **Check for anomalies** — Accounts without recent access, SOD violations, or entitlement creep
- **Certify (Approve)** — If the access is still needed
- **Revoke** — If the access is no longer needed, with a reason

### Add Comments (Optional)
Add comments for any revocation decision — this is critical for audit purposes and potential user appeals.

### Submit the Campaign
After reviewing all items, click **Submit** or **Finish**. The results are processed, and revocations are sent to the provisioning system for execution.
</Steps>

![SailPoint certification review detail page showing individual user entitlements with Certify and Revoke buttons and justification fields](https://community.sailpoint.com/t5/image/serverpage/image-id/2906iA1900F9E36D10D3E/image-size/large?v=v2&px=999)

### 4.2 Creating an Access Request Policy

Access request policies define **who can request what access**, **from whom they need approval**, and **how long the access is granted** — similar to PAM's elevation workflows.

<Steps>
### Open Access Request Configuration
Go to **Access Request** → **Policies** → **New Policy**.

### Define the Access Item
Select the application, entitlement, or role that users can request (e.g., VPN access to production, or a specific Active Directory security group).

### Set Approval Chain
Configure one or more approval stages:
- **Level 1** — Manager approval (always)
- **Level 2** — Application owner approval (for sensitive entitlements)
- **Level 3** — Security team approval (for privileged access)

### Set Duration
Define a default access duration (e.g., 30 days for project access). After expiry, SailPoint automatically triggers a revocation request.

### Enable Automatic Provisioning
Connect the policy to a provisioning connector (Active Directory, Azure AD, AWS, etc.) so approved requests are automatically provisioned without manual intervention.
</Steps>

### 4.3 Running a Role Mining Model

Role mining analyses existing user entitlements to suggest optimal role definitions — a core governance capability.

<Steps>
### Navigate to Role Mining
Go to **Roles** → **Role Mining** → **New Role Mining Model**.

### Define the Scope
Select the source application (e.g., Active Directory, SAP) and the user population (e.g., all Users, or a subset like Finance department).

### Run the Model
Select the algorithm parameters:
- **Coverage** — What % of users should the roles cover
- **Granularity** — How many roles to generate (fewer = broader, more = precise)
- **Exclusion rules** — Accounts to exclude from analysis (e.g., break-glass accounts)

### Review Role Candidates
The model returns suggested role definitions with:
- **Proposed role name** and description
- **User count** — How many users would be in each role
- **Entitlement list** — The permissions included
- **Coverage percentage** — How well the role model fits the current state

### Publish Roles
Review and select role candidates, then publish them as **provisional roles** for managerial review and eventual activation.
</Steps>

<Aside variant="info">
SailPoint is **not a PAM platform** — it governs *who has access to what* and enforces separation of duties. PAM handles *how that access is used and monitored*. They complement each other: SailPoint defines the policy, PAM enforces it at runtime.
</Aside>

## BeyondTrust Privileged Access Management

BeyondTrust (formerly BeyondTrust PowerBroker) provides endpoint privilege management, credential vaulting, and session management. It is particularly strong for **Windows and Linux endpoint least privilege**.

### 5.1 Creating a Privilege Elevation Rule

![BeyondTrust Management Console showing the Privilege Elevation policy editor with rule conditions and allowed actions](https://www.beyondtrust.com/docs/privileged-access-management/images/pam-rule-policy.png)

**Steps:**

<Steps>
### Open the Management Console
Launch the BeyondTrust **Management Console** (cloud or on-premises) and navigate to **Policies** → **Privilege Elevation**.

### Create a New Rule
Click **Add Rule** and give it a descriptive name (e.g., `Allow_Approved_Patches_Install`).

### Define Target Conditions
Set the conditions for when this rule applies:
- **Computer groups** — Apply to specific groups (e.g., All Windows Servers)
- **User groups** — Apply to specific user groups (e.g., Server_Admins)
- **File path** — The application or script being elevated (e.g., `C:\Windows\System32\msiexec.exe`)
- **File hash** — Optional: validate file hash for integrity

### Define Allowed Actions
Configure what elevated actions are permitted:
- **Run with elevated token** — Full administrator elevation
- **Run as specific user** — Run as a managed service account
- **Suppress UAC prompts** — Silent elevation without user notification

### Set Policy Enforcement
Choose enforcement mode:
- **Automatic** — Elevation happens silently when conditions match
- **Approval required** — User must request and receive approval
- **Justification required** — User must provide a reason for elevation
</Steps>

![BeyondTrust elevation request dialog showing the user interface for requesting privilege elevation with justification field](https://www.beyondtrust.com/docs/privileged-access-management/images/pam-endpoint-request.png)

### 5.2 Managing Application Groups

Application groups define which applications can be elevated and how.

<Steps>
### Navigate to Application Groups
In the Management Console, go to **Application Groups** → **Add Group**.

### Define the Application
Configure:
- **Name** — e.g., "Approved Software Installers"
- **File path rules** — `*.msi`, `*.exe`, or specific paths
- **Publisher rules** — Allow all applications signed by a specific publisher (e.g., Microsoft Corporation)
- **Hash rules** — Pin specific versions by SHA-256 hash

### Set Elevation Behaviour
- **Elevation type** — Full, restricted, or user-context
- **UI behaviour** — Silently elevate, show prompt, or require approval

### Assign to Policy
Link this application group to a policy created in the previous section.
</Steps>

### 5.3 Configuring Credential Vaulting in BeyondTrust

<Steps>
### Navigate to Credential Vault
Go to **Vault** → **Managed Accounts** → **Add Account**.

### Enter Account Details
- **Account name** — e.g., `SRV-DB01_LocalAdmin`
- **Target system** — Hostname or IP address
- **Credential type** — Password, SSH key, or API token
- **Checkout policy** — Require approval, allow self-service, or require MFA

### Set Rotation Schedule
- **Interval** — Every 24 hours, 7 days, 30 days, or custom
- **Rotation type** — Automatic on check-in, periodic, or manual
- **Password complexity** — Set length, special characters, and character classes

### Configure Session Recording
Enable session recording for this account. BeyondTrust supports:
- **RDP session recording** — Full video with text extraction
- **SSH session recording** — Command logging with search
- **Database session proxy** — SQL query logging

### Save and Test
Click **Save** and then **Test Connection** to verify that BeyondTrust can reach the target and rotate the credential.
</Steps>

<Aside variant="tip">
BeyondTrust's strength is **endpoint privilege management** — removing local admin rights from workstations while allowing users to perform approved elevated tasks. This is the most direct implementation of least privilege for end-user devices.
</Aside>

## Quick Reference: Common Task Matrix

| I Need To... | CyberArk | Entra ID PIM | AWS IAM | SailPoint | BeyondTrust |
|-------------|----------|-------------|---------|-----------|-------------|
| Store a privileged password | Add Account to Safe | ❌ | Secrets Manager | ❌ | Add Managed Account |
| Elevate my access temporarily | Check out account | Activate PIM role | Assume IAM role | Request access | Request elevation |
| Record an admin session | Connect via PSM | ❌ | CloudTrail (API only) | ❌ | Enable session recording |
| Review who has admin access | Vault reports | Access Reviews | IAM Credential Report | Access certification | Policy reports |
| Remove local admin rights | ❌ | ❌ | ❌ | Role-based provisioning | Application control & elevation |
| Approve a colleague's elevation | Vault approval | PIM approval workflow | ❌ | Access request approval | Elevation approval |
| Automate password rotation | CPM policy | ❌ | Secrets Manager rotation | Connector provisioning | Rotation schedule |

<Aside variant="caution">
Each of these platforms has evolved significantly. Always refer to the **vendor's official documentation** for the most current interface layout and feature availability. The principles covered here (vaulting, JIT, least privilege, certification, session recording) remain consistent across platforms, even if UI paths vary.
</Aside>
